# Обработчик очередей Playwright

## Актуальная архитектура

Контейнер Playwright запускает основное Node.js-приложение Competitor Analyzer.

Актуальный код приложения находится в корневом каталоге:

```text
src/
```

Внутри контейнера корень проекта подключён по пути:

```text
/app
```

Поэтому актуальные исходные файлы доступны внутри контейнера по пути:

```text
/app/src/
```

## Управление worker

Запуск worker определяется переменными окружения:

```text
ENABLE_ANALYSIS_WORKER
ENABLE_SCRAPING_WORKER
ENABLE_LEGACY_WORKER
```

Подтверждённая конфигурация на 22 июля 2026 года:

```text
ENABLE_ANALYSIS_WORKER=true
ENABLE_SCRAPING_WORKER=false
ENABLE_LEGACY_WORKER=false
```

Следовательно:

- `Analysis Worker` включён;
- новый `Scraping Worker` отключён;
- устаревший Playwright worker отключён.

## Интервал опроса очереди

Подтверждённый интервал опроса:

```text
15000 миллисекунд
```

Свободный worker проверяет очередь примерно один раз в 15 секунд.

## Универсальный цикл worker

Файл:

```text
src/workers/base-worker.js
```

Он отвечает за общий цикл работы:

1. выполняет действие перед очередным циклом, если оно настроено;
2. получает следующее задание;
3. ожидает, если заданий нет;
4. записывает краткую информацию о полученном задании;
5. передаёт задание обработчику;
6. перехватывает ошибку цикла;
7. продолжает работу после ошибки;
8. корректно завершает цикл после сигнала остановки.

## Analysis Worker

Файл:

```text
src/workers/analysis-worker.js
```

Analysis Worker:

1. проверяет наличие зависших аналитических заданий;
2. восстанавливает зависшие задания;
3. получает следующее задание из `analysis_jobs`;
4. передаёт задание в Analysis Service;
5. сохраняет успешный результат или ошибку;
6. продолжает опрашивать очередь.

## Работа с очередью

Модуль очереди:

```text
src/queue/analysis-queue.js
```

Для получения задания используется PostgreSQL-функция:

```text
take_next_analysis_job()
```

Для безопасной параллельной обработки применяется:

```text
FOR UPDATE SKIP LOCKED
```

Полученное задание переводится из статуса `new` в статус `running`.

Для успешного завершения используется функция:

```text
complete_analysis_job()
```

Результат сохраняется в поле:

```text
analysis_jobs.result_json
```

Для завершения с ошибкой используется функция:

```text
fail_analysis_job()
```

Для возврата зависших заданий в очередь используется функция:

```text
recover_stuck_analysis_jobs()
```

## Маршрутизация анализа

Файл маршрутизатора:

```text
src/services/analysis-router.js
```

Поддерживаются типы анализа:

```text
site_structure
seo_analysis
```

Маршруты обработки:

```text
site_structure -> analyzeStructure()
seo_analysis   -> analyzeSeo()
```

## Компактное логирование

Ранее универсальный worker записывал в журнал полный объект задания.

В журнал могли попадать большие поля:

```text
page_text
page_structure
possibleProductCards
```

Теперь `base-worker.js` поддерживает функцию:

```text
summarizeJob()
```

Analysis Worker передаёт в журнал только следующие поля:

```text
analysisJobId
pageSnapshotId
competitorSiteId
analysisType
pageTitle
finalUrl
```

Это уменьшает размер Docker-логов и исключает повторное сохранение полного содержимого снимка страницы в журнале.

## Подтверждённая проверка

На задании:

```text
analysis_jobs.id = 12
```

подтверждено:

- задание успешно получено Analysis Worker;
- обработка завершилась со статусом `completed`;
- полный `page_text` отсутствует в записи получения задания;
- полный `page_structure` отсутствует в записи получения задания;
- компактные поля задания присутствуют в журнале.

## Устаревшая реализация

Предыдущие worker могут сохраняться в каталоге:

```text
playwright/src/
```

Их запуск отключён:

```text
ENABLE_LEGACY_WORKER=false
```

Эти файлы не являются актуальным обработчиком очереди `analysis_jobs`.

## Технический долг

Необходимо дополнительно реализовать или проверить:

- максимальное количество повторных попыток;
- разделение временных и постоянных ошибок;
- метрики времени обработки;
- метрики длины очереди;
- ограничение размера одного задания;
- завершение активного задания при остановке контейнера;
- независимое масштабирование разных worker.
