# Обзор схемы PostgreSQL

## Общая логика данных

```text
clients
└── competitor_sites
    ├── scan_jobs
    │   └── page_snapshots
    │       └── analysis_jobs
    ├── scraping_rules
    │   └── product_scan_jobs
    │       └── price_history
    ├── products
    │   └── price_history
    └── site_maps
```

## Основной поток исследования сайта

```text
clients
→ competitor_sites
→ scan_jobs
→ page_snapshots
→ analysis_jobs
```

1. Создаётся клиент.
2. Для клиента добавляется сайт конкурента.
3. В `scan_jobs` создаётся задание исследования.
4. Playwright сохраняет снимок в `page_snapshots`.
5. Создаётся GPT-задание в `analysis_jobs`.
6. GPT сохраняет результат анализа.

## Основной поток сбора товаров

```text
competitor_sites
→ scraping_rules
→ product_scan_jobs
→ products
→ price_history
```

1. Для сайта создаётся правило сбора.
2. Создаётся задание `product_scan_jobs`.
3. Playwright открывает страницы каталога.
4. Товары создаются или обновляются в `products`.
5. Каждая полученная цена сохраняется в `price_history`.

## JSONB-поля

В проекте используются JSONB-поля:

```text
page_snapshots.page_structure
analysis_jobs.result_json
site_maps.map_json
```

### page_structure

Хранит структурированное представление страницы:

- заголовки;
- ссылки;
- кнопки;
- поля ввода;
- изображения;
- возможные карточки товаров;
- статистику элементов.

Для поля создан GIN-индекс:

```text
idx_page_snapshots_page_structure_gin
```

### result_json

Хранит структурированный результат анализа GPT.

### map_json

Предназначено для хранения карты сайта.

На момент инвентаризации таблица `site_maps` не содержит записей.

## Типы времени

Прикладные таблицы используют:

```text
timestamp without time zone
```

Это означает, что часовой пояс в самом значении не сохраняется.

Приложение и документация должны использовать единое соглашение о времени. Рекомендуется хранить время в UTC и явно преобразовывать его при отображении.

## Идентификаторы

Основные таблицы используют:

```text
SERIAL
```

Фактически PostgreSQL создаёт:

- поле типа `integer`;
- отдельную sequence;
- значение по умолчанию через `nextval(...)`.

В будущих миграциях можно перейти на стандартный синтаксис:

```sql
GENERATED BY DEFAULT AS IDENTITY
```

Но это не является срочной задачей.
